# La estrategia de ejecución se basa en la dependencia funcional entre los módulos y en el principio de que una falla en capas básicas invalida cualquier resultado de capas superiores. El orden es el siguiente:

# **Nivel 1 — Prioridad CRÍTICA: Pruebas Unitarias de Caja Blanca (TW-01 a TW-10)**

**Tipo:** Unitarias y Nivel Caja Blanca

**Objetivo**: validar la lógica interna del backend antes de pasar a pruebas de interfaz.

**Criterio**: si falla la lógica básica, no se ejecutan los flujos superiores.

**Justificación: **Estas pruebas se ejecutan primero porque validan la lógica interna del backend, que es el motor de todo el sistema. Si el método \`validarUsuario\` tiene un error lógico en sus condicionales, entonces la prueba de login en la interfaz fallará, pero la causa raíz no será el login, sino el método interno. Al probar unitariamente primero, se aisla el origen de cada defecto y se evita perder tiempo investigando fallas en la UI que en realidad provienen de un bug en el código. Por ejemplo, si \`calcularCalificacionJuego\` (TW-08) divide entre cero cuando no hay reseñas, todas las pruebas del detalle de juego que muestran una calificación en pantalla serán inválidas hasta corregirlo.

**Nivel 2 — Prioridad ALTA: Control de Acceso (Registro y Login)**

**Casos: **TB-01, TB-02, TB-03, TB-04

**Tipo:** Sistema y Aceptación y Nivel: Caja Negra

**Objetivo:** asegurar el punto de entrada al sistema.

**Criterio:** sin usuario autenticado no se puede probar el resto del sistema.

**Justificación: **El registro y el login son los únicos puntos de entrada al sistema. Sin un usuario autenticado, ninguna otra funcionalidad (listas personales, reseñas, perfil) puede ser accedida ni probada. Son funcionalidades bloqueantes: un fallo aquí detiene el avance de la totalidad del plan de pruebas. Por eso se ejecutan como primera prioridad dentro de las pruebas de interfaz, y se deben ejecutar inmediatamente después de confirmar que la lógica interna (Caja Blanca) está correcta.

**Nivel 3 — Prioridad ALTA: Integración del Perfil de Usuario con Persistencia**

**Casos:** TG-01 Tipo: Integración y Nivel:Caja Gris

**Objetivo:** confirmar que el usuario registrado se guarda y puede volver a iniciar sesión.

**Criterio:** verificar usuarios.json o el backend y el flujo login.

**Justificación: **Antes de probar cualquier funcionalidad que dependa de datos del usuario (listas, reseñas, perfil), es necesario confirmar que el ciclo completo de registro-persistencia-login funciona de extremo a extremo. TG-01 verifica que el usuario registrado realmente existe en \`usuarios.json\` y que el sistema lo recupera correctamente en el login. Si este flujo de integración falla, todos los casos siguientes que asuman que hay un usuario activo en sesión perderían validez.

** Nivel 4 — Prioridad MEDIA-ALTA: Funcionalidades Principales del Negocio (Catálogo y Listas)**

# **Casos: ** TB-07, TB-08, TB-10, TB-11, TB-12, TG-04, TG-05

# **Tipo**: Sistema e Integración y Nivel: Caja Negra y Gris

**Objetivo:** validar catálogo y listas personales.

**Criterio: **el usuario debe poder ver juegos y gestionarlos.

# **Justificación: ** El catálogo y las listas personales representan el valor central del producto GamingList: que el usuario pueda ver juegos y organizarlos. Se priorizan antes que las reseñas porque son la funcionalidad de lectura y gestión básica. La integración TG-04 y TG-05 se incluyen en esta etapa porque validan que los datos cruzados del catálogo y el perfil se muestren correctamente, lo que es visible para el usuario en su uso cotidiano.


# **Nivel 5 — Prioridad MEDIA: Reglas de Negocio Complejas (Reseñas, Límites y Cascada)**

# **Casos: **TB-05, TB-06, TB-13, TB-14, TB-15, TG-02, TG-03, TG-06

# **Tipo:** Sistema e Integración y Nivel: Caja Negra y Gris

**Objetivo:** validar límites de negocio como reseñas duplicadas, promedio de calificaciones, eliminación en cascada.

**Criterio: **la integridad de los datos debe mantenerse.

# **Justificación: **Estas pruebas validan reglas de negocio más específicas: los límites de año de un videojuego, el bloqueo de doble reseña, el cálculo correcto del promedio de calificaciones, y la eliminación en cascada. Son importantes porque protegen la integridad de los datos, pero su fallo no impide que el usuario use el sistema básico. Por ejemplo, que un usuario pueda escribir dos reseñas del mismo juego es un defecto funcional pero no bloquea el acceso al catálogo ni al perfil.


# **Nivel 6 — Prioridad BAJA: Casos de Borde y Comportamiento Secundario**

# **Casos:** TB-09, TG-05 (aspecto visual)

# **Tipo: **Sistema y Nivel: Caja Negra

**Objetivo:** comprobar manejo de errores y rutas no válidas.

**Criterio:** el sistema no debe colapsar ante entradas inválidas.

# **Justificación: ** La prueba de enrutamiento de error (TB-09) verifica que el sistema no colapse ante una URL inválida, mostrando \`ErrorView\`. Es un comportamiento defensivo importante pero de baja frecuencia de uso. Se ejecuta al final porque no depende de ningún dato de usuario ni del catálogo, y su falla no afecta ningún flujo de negocio principal.
